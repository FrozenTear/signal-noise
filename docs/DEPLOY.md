# Deploying Signal Noise to the VPS

Direct hosting on the board's VPS (`194.163.163.153`, SSH user `ts6dev`).
**No GitHub Actions, no git pull on the box, no container registry** — build
locally, ship the artefacts with `rsync`, run under `systemd`, front with Caddy.
Approved design: [THE-47 plan](/THE/issues/THE-47#document-plan). Implementation
tracked on [THE-114](/THE/issues/THE-114).

## Architecture notes (read first)

- **SurrealDB is embedded** (`engine::local::SurrealKv`, `src/api/db.rs`) — a
  *file* at `data/signal-noise.db`, **not** a separate DB service. There is no
  `surreal start` daemon to run. "Persistent storage" = the working directory
  `/var/lib/ainory-times/data` must survive deploys. The schema is compiled into
  the binary via `include_str!`, so it self-applies on boot.
- **Single writer.** The embedded store is single-writer. The offline tools
  (`db_admin`, `seed_*`, `read_article`) open the same file and must only run
  while the `ainory-times` service is **stopped**.
- **The `scanner` bin does not touch the DB** — it reads RSS/gnews and posts to
  the Paperclip API, so it can run independently (see "Scanner" below).

## Layout on the VPS

| Path | Purpose |
|---|---|
| `/opt/ainory-times/releases/<ts>/` | immutable release dirs (rsync target) |
| `/opt/ainory-times/current` | symlink → active release |
| `/var/lib/ainory-times/data/` | embedded SurrealKV DB — **never** rsynced |
| `/etc/systemd/system/ainory-times.service` | app service unit |
| `/etc/caddy/Caddyfile` | reverse proxy → `127.0.0.1:8888` |
| `/etc/ainory-times/scanner.env` | (optional) Paperclip creds for the scanner |

App runs as the `ainory` system user, bound to `127.0.0.1:8888`, Caddy fronts it.

## One-time bring-up

```sh
export AINORY_VPS_HOST=194.163.163.153    # SSH user defaults to ts6dev
# export AINORY_DOMAIN=news.example.com   # set for auto-TLS; omit for IP-only HTTP

./scripts/inspect.sh      # read-only: OS, RAM/disk, ports, firewall, sudo
./scripts/provision.sh    # create ainory user + dirs, install Caddy, drop units
./scripts/deploy.sh       # build locally, ship, symlink-swap, restart, health-check
```

`provision.sh` requires `sudo` for `ts6dev`. If `ts6dev` lacks sudo, the board
must either grant it or run the provisioning one-liners manually (they are echoed
in the script).

## Routine deploys

```sh
AINORY_VPS_HOST=194.163.163.153 ./scripts/deploy.sh
```

Builds with `dx bundle --release`, rsyncs to a fresh timestamped release dir,
atomically swaps `current`, restarts the service, prunes to the last 5 releases,
and health-checks `http://127.0.0.1:8888/` on the box.

**First-deploy check:** confirm the exact path of the server binary inside the
`dx bundle` output. The unit's `ExecStart` assumes
`/opt/ainory-times/current/signal-noise`; if `dx` nests it (e.g. under
`server/`), update `ExecStart` in `deploy/ainory-times.service` and re-run
`provision.sh` once.

## Rollback

```sh
AINORY_VPS_HOST=194.163.163.153 ./scripts/rollback.sh
```

Flips `current` to the previous release dir and restarts. Releases are immutable
and the data dir is untouched, so rollback is safe.

## Scanner (optional, off by default)

The scanner is a separate binary (`--bin scanner`, not part of the `dx bundle`).
To run it on the VPS:

1. Build + ship it: `cargo build --release --features server --bin scanner` then
   rsync `target/release/scanner` into `/opt/ainory-times/current/`.
2. Create `/etc/ainory-times/scanner.env` (`root:ainory`, `0640`) with
   `PAPERCLIP_API_URL`, `PAPERCLIP_API_KEY`, `PAPERCLIP_COMPANY_ID`.
3. `sudo cp deploy/scanner.{service,timer} /etc/systemd/system/ && sudo systemctl enable --now scanner.timer`

Without `scanner.env` the timer is inert (`EnvironmentFile=-` tolerates the
missing file). The scanner does **not** open the embedded DB, so it is safe to
run alongside the app service.
